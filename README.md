# AIT-ADS Parser

A Python package that ingests the **AIT Alert Data Set** (AIT-ADS, Zenodo record 8263181) and normalises its three different JSON formats (Wazuh native, Suricata-via-Wazuh, AMiner) into a single unified schema suitable for downstream LLM/RAG triage research and traditional ML baselines.

This is the dataset preparation layer for an MRes Cybersecurity dissertation on **LLM-based intelligent alert triage and prioritisation for backend microservice security operations**.

## 1. What This Package Does

The AIT-ADS dataset is a collection of **2,655,821 security alerts** generated by three different intrusion detection systems (Wazuh, Suricata, AMiner) running over simulated multi-step attack scenarios. The alerts arrive in **three different JSON schemas** spread across **two file types** (`*_aminer.json` and `*_wazuh.json`), with the Wazuh files containing a *mix* of native Wazuh alerts and Suricata alerts wrapped in a Wazuh envelope.

This package solves four problems:

1. **Format heterogeneity** — three IDS, three schemas. Downstream code shouldn't have to deal with that.
2. **Severity inconsistency** — Wazuh uses 0–16 (higher = worse), Suricata uses 1–3 (lower = worse), AMiner has no severity at all. Normalised here to a unified 1–5 scale.
3. **Dual-purpose output** — the LLM/RAG pipeline wants nested JSON; traditional ML baselines want flat tables. Both are produced from one parse.
4. **Validation** — confirms the parsed counts match Zenodo's published totals so silent data loss is caught immediately.

After running this package, your dataset is ready for labelling, knowledge base construction, and experimental evaluation.

## 2. Why AIT-ADS

AIT-ADS is the primary dataset for this dissertation because:

- It contains **alerts**, not raw logs. No additional IDS step required.
- The alerts are produced by **production-grade tools** (Wazuh, Suricata, AMiner), so realistic false positives are baked in.
- It covers **eight scenarios** with **multi-step attack chains** (network scans → service scans → web exploitation → cracking → privilege escalation → exfiltration), supporting alert correlation evaluation.
- It's **less saturated in the LLM literature** than HDFS or CICIDS2017, providing genuine novelty headroom.
- The licence is **CC-BY 4.0** (attribution only), the most permissive academic licence.

For a full comparison of why this was chosen over HDFS, CICIDS2017, and AIT-LDS v2, see the journey blog post (`dissertation_dataset_journey.md`).

## 3. Background: The Three IDS Formats

Before running the parser, it helps to understand what's actually in the files.

### 3.1 Wazuh native alerts

Wazuh is a host-based intrusion detection system (HIDS) that processes log files (syslog, audit, application logs) and matches them against rules. A native Wazuh alert in AIT-ADS looks like:

```json
{
  "predecoder": {"hostname": "mail", "program_name": "freshclam"},
  "agent": {"ip": "172.17.131.81", "name": "wazuh-client", "id": "18"},
  "rule": {
    "level": 3,
    "description": "ClamAV database update",
    "groups": ["clamd", "freshclam", "virus"],
    "id": "52507",
    "pci_dss": ["5.2"],
    "nist_800_53": ["SI.3"],
    "gdpr": ["IV_35.7.d"]
  },
  "decoder": {"name": "freshclam"},
  "full_log": "Jan 15 02:32:32 mail freshclam[29266]: ...",
  "@timestamp": "2022-01-15T02:32:32.000000Z",
  "location": "/var/log/syslog"
}
```

Key fields the parser uses: `@timestamp`, `rule.level`, `rule.description`, `rule.id`, `rule.groups`, `rule.pci_dss`, `rule.nist_800_53`, `rule.gdpr`, `agent.name`, `agent.ip`, `predecoder.program_name`, `full_log`, `location`.

### 3.2 Suricata-via-Wazuh alerts

Suricata is a network-based IDS (NIDS) that inspects traffic flows. In AIT-ADS, Suricata alerts are forwarded through Wazuh, which wraps them in its own envelope. They live inside the same `*_wazuh.json` files as native Wazuh alerts:

```json
{
  "agent": {"ip": "10.35.32.1", "name": "wazuh-client", "id": "29"},
  "data": {
    "src_ip": "192.168.128.4",
    "src_port": "56852",
    "event_type": "alert",
    "alert": {
      "severity": "3",
      "signature_id": "2013504",
      "signature": "ET POLICY GNU/Linux APT User-Agent Outbound...",
      "category": "Not Suspicious Traffic"
    },
    "dest_ip": "91.189.95.85",
    "dest_port": "80",
    "proto": "TCP"
  },
  "rule": {"level": 3, "groups": ["ids", "suricata"], "id": "86601"},
  "@timestamp": "2022-01-15T03:45:39.681006+0000"
}
```

The dispatcher detects these by looking for `data.event_type == "alert"` AND `data.alert.signature` being present. **Critical**: Suricata's severity scale is **inverted** compared to Wazuh's — `1` is most severe, `3` is least.

### 3.3 AMiner anomaly alerts

AMiner is an anomaly-based detector — it learns a baseline and flags deviations. It has no rule, no severity, no signature:

```json
{
  "AnalysisComponent": {
    "AnalysisComponentType": "NewMatchPathDetector",
    "Message": "New path(es) detected",
    "TrainingMode": true
  },
  "LogData": {
    "RawLogData": ["type=USER_ACCT msg=audit(1642204801.159:657): ..."],
    "Timestamps": [1642204801.16],
    "DetectionTimestamp": [1642204801.16],
    "LogResources": ["/var/log/audit/audit.log"]
  },
  "AMiner": {"ID": "172.17.129.140"}
}
```

AMiner uses CamelCase keys (different from Wazuh's snake_case), wraps scalars in arrays (`Timestamps[0]`), uses Unix epoch floats for timestamps, and identifies hosts by IP. The `TrainingMode` flag matters: while AMiner is still learning the baseline, every "anomaly" is essentially false-positive noise.

## 4. Prerequisites

- **Python 3.10 or higher** (uses modern type hints like `int | None`)
- **Ubuntu/Linux/macOS** (developed on Ubuntu; Windows should work but isn't tested)
- **At least 8 GB free disk space** for raw data + processed outputs
- **At least 4 GB RAM** for the full ingestion (it streams, but holds flat rows in memory before writing)

The parser core uses only the Python standard library — **no required dependencies**.

For the parquet output (recommended, much faster and smaller than CSV at 2.6M rows), you need:

```bash
pip install pyarrow
```

After the full ingestion, `data/processed/` contains four files:

One UnifiedAlert per line in JSON format, **including the original `raw_record`** for downstream RAG embedding. This is the primary input for the LLM/RAG pipeline.

Inspect a record:

```bash
head -1 data/processed/alerts.jsonl | python3 -m json.tool | head -40
```

### 9.2 `alerts.parquet` or `alerts.csv` (~300 MB parquet, ~2 GB CSV)

Flat columnar format **without `raw_record`**, with list fields joined by semicolons. This is the input for the rule-based baseline, Random Forest, SVM, and any other tabular ML approach.

Inspect with pandas:

```python
import pandas as pd
df = pd.read_parquet("data/processed/alerts.parquet")
print(df.shape)
print(df.head())
print(df["source_ids"].value_counts())
print(df["severity_norm"].value_counts())
```

### 9.3 `stats.json` (~5 KB)

Summary statistics including:

- `total_written` — should equal 2,655,821
- `by_source` — counts per IDS
- `by_scenario_source` — counts per (scenario, IDS) pair
- `by_severity_norm` — distribution across the 1–5 normalised severity scale
- `output_format` — "parquet" or "csv"
- `validation` — Zenodo count check
- `expected_totals` — what we expected
- `scenarios_ingested` — which scenarios were processed

These numbers go directly into the dissertation methodology section.

### 9.4 `parse_errors.jsonl` (should be empty)

Records that failed to parse, with file path, line number, and error reason. In a clean run this file should be empty or contain only a handful of entries (out of 2.6M, that's noise). If it has thousands of entries, investigate.

## 10. The Unified Schema

All three parsers produce instances of the `UnifiedAlert` dataclass defined in `parsers/unified_alert.py`. Every field with full reasoning:

| Field | Type | Source | Purpose |
|---|---|---|---|
| `alert_id` | `str` | Generated UUID | Stable unique ID for cross-referencing |
| `timestamp` | `datetime` (UTC) | `@timestamp` (Wazuh) / `data.timestamp` (Suricata) / `LogData.DetectionTimestamp[0]` (AMiner) | When the alert was raised |
| `source_ids` | `str` | Constant per parser | "wazuh" / "suricata" / "aminer" |
| `scenario` | `str` | Filename prefix | One of the 8 AIT scenarios |
| `host` | `str \| None` | `agent.name` (Wazuh/Suricata) / `AMiner.ID` (AMiner) | Source host identifier |
| `host_ip` | `str \| None` | `agent.ip` / `AMiner.ID` | Source host IP |
| `log_source` | `str \| None` | `location` (Wazuh/Suricata) / `LogData.LogResources[0]` (AMiner) | Path of the log file that triggered |
| `program_name` | `str \| None` | `predecoder.program_name` | Wazuh-only; the program that wrote the log |
| `severity_raw` | `int \| None` | `rule.level` (Wazuh) / `data.alert.severity` (Suricata) | Native severity before normalisation |
| `severity_norm` | `int` (1–5) | Computed | Unified severity scale, see Section 11 |
| `rule_id` | `str \| None` | `rule.id` / `data.alert.signature_id` / `AnalysisComponentIdentifier` | Native rule identifier |
| `description` | `str` | `rule.description` / `data.alert.signature` / `AnalysisComponent.Message` | Human-readable summary |
| `raw_message` | `str` | `full_log` / `data.alert.category` / `LogData.RawLogData[0]` | Original log line |
| `src_ip` | `str \| None` | `data.srcip` / `data.src_ip` | Source IP (mainly Suricata) |
| `dst_ip` | `str \| None` | `data.dstip` / `data.dest_ip` | Destination IP |
| `src_port` | `int \| None` | `data.src_port` | Source port |
| `dst_port` | `int \| None` | `data.dest_port` | Destination port |
| `protocol` | `str \| None` | `data.proto` | TCP / UDP / etc. |
| `rule_groups` | `list[str]` | `rule.groups` (Wazuh/Suricata) / `[detector_name]` (AMiner) | Rule classification tags |
| `mitre_techniques` | `list[str]` | `rule.mitre.id` | MITRE ATT&CK technique IDs |
| `mitre_tactics` | `list[str]` | `rule.mitre.tactic` | MITRE ATT&CK tactic names |
| `pci_dss` | `list[str]` | `rule.pci_dss` | PCI-DSS compliance mappings |
| `nist_800_53` | `list[str]` | `rule.nist_800_53` | NIST 800-53 controls |
| `gdpr` | `list[str]` | `rule.gdpr` | GDPR articles |
| `aminer_detector` | `str \| None` | `AnalysisComponent.AnalysisComponentType` | AMiner detector name |
| `aminer_training_mode` | `bool` | `AnalysisComponent.TrainingMode` | True if AMiner is still learning baseline |
| `is_attack` | `bool` | Filled by labeller (NOT the parser) | Whether alert falls within an attack window |
| `attack_phase` | `str \| None` | Filled by labeller | Which phase from `labels.csv` |
| `raw_record` | `dict` | Original parsed JSON | Full record preserved for RAG embedding |

The two output methods:

- `to_jsonl_dict()` — preserves `raw_record`, used for `alerts.jsonl`
- `to_flat_dict()` — drops `raw_record`, joins lists with `;`, used for parquet/CSV

## 11. Severity Normalisation Explained

The single most subtle piece of logic in this parser. Defined in `parsers/severity_mapping.py`.

### Target scale (unified 1–5)

| Level | Meaning |
|---|---|
| 1 | Informational / training-mode anomaly |
| 2 | Low (e.g., routine policy notice) |
| 3 | Medium (suspicious but not confirmed malicious) |
| 4 | High (likely malicious) |
| 5 | Critical (confirmed exploit, privilege escalation, exfiltration) |

### Wazuh: 0–16 → 1–5

Wazuh's `rule.level` ranges 0–16 where higher = more severe (per official documentation):

```
level 0–3   → 1
level 4–6   → 2
level 7–9   → 3
level 10–12 → 4
level 13–16 → 5
```

### Suricata: 1–3 (inverted) → 1–5

**Critical**: Suricata uses 1–3 where **lower = more severe** (the opposite of Wazuh):

```
severity 1 → 5  (critical, e.g., admin privilege gain)
severity 2 → 4  (high)
severity 3 → 2  (low / policy-style)
```

If you write a parser without inverting this, every Suricata severity will be wrong. The unit tests in `test_parsers.py` specifically verify the inversion.

### AMiner: TrainingMode flag → 1 or 3

AMiner has no native severity. The defensible default:

```
TrainingMode == True  → 1  (still learning baseline; alerts are noise)
TrainingMode == False → 3  (real anomaly outside learned baseline)
```

Analysts can override per detector type in downstream analysis if needed. The mapping is auditable (single named module) and explicitly justified in the dissertation methodology.

## 12. How the Dispatcher Works

The `_wazuh.json` files contain **two** alert formats interleaved. The dispatcher in `ingest.py` (`iter_wazuh_file`) routes each line to the correct parser:

```python
def is_suricata_record(record: dict) -> bool:
    data = record.get("data") or {}
    if data.get("event_type") != "alert":
        return False
    alert = data.get("alert")
    if not isinstance(alert, dict):
        return False
    return "signature" in alert
```

Both conditions must hold. Either alone would produce false positives:

- Native Wazuh records can have `data.event_type` set to other values (and often have no `data.alert` at all).
- Some Wazuh decoders produce records with `data.alert` but no `signature`.

If `is_suricata_record()` returns true → dispatch to `suricata_parser.parse_record()`.
Otherwise → dispatch to `wazuh_parser.parse_record()`.

## 13. Validation Against Zenodo Counts

Zenodo publishes the per-source totals for AIT-ADS:

| Source | Expected count |
|---|---|
| Wazuh | 2,293,628 |
| Suricata | 306,635 |
| AMiner | 55,558 |
| **Total** | **2,655,821** |

The orchestrator validates against these automatically. The `stats.json` file includes a `validation` block:

```json
{
  "validation": {
    "matches_zenodo": true,
    "discrepancies": []
  }
}
```

If counts don't match, `discrepancies` will list each mismatched source with `expected`, `got`, and `diff`. This is the **single most important sanity check** — silent data loss in a 2.6M-record pipeline is otherwise extremely hard to detect.

Validation only runs when `--scenarios` is not specified and `--limit` is not set (i.e., a full ingestion).

## 14. Common Issues and Troubleshooting

### 14.1 `ImportError: No module named 'parsers'`

Cause: running `ingest.py` from inside the `ait_parser/` directory while imports are relative.

Fix: run from the project root, with the path to `ingest.py`:

```bash
# Correct
python3 ait_parser/ingest.py --input data/ait-ads --output data/processed

# Also works (cd into ait_parser first)
cd ait_parser && python3 ingest.py --input ../data/ait-ads --output ../data/processed
```

### 14.2 `ModuleNotFoundError: No module named 'pyarrow'`

Cause: pyarrow not installed. The parser falls back to CSV automatically — this is a soft failure, not an error.

Fix (optional, for parquet): `pip3 install pyarrow --break-system-packages`

### 14.3 Counts don't match Zenodo

Cause: usually one of:

- **Incomplete download.** Re-run `wget` for the affected file. Verify with `wc -l`.
- **Parser dropping records silently.** Check `parse_errors.jsonl` — if it has many entries, investigate the specific error messages.
- **Encoding issues.** AIT-ADS files are UTF-8; if your system locale is different, `open(path, "r")` may fail. Try `open(path, "r", encoding="utf-8")`.

### 14.4 Out of memory during full ingestion

Cause: `flat_rows` list accumulates 2.6M dicts in memory before writing the parquet/CSV.

Fix: stream the flat output too. Modify `write_outputs()` to write rows in batches (e.g., flush every 100k records). For 2.6M rows of moderately-sized dicts you typically need 4–6 GB RAM; if your machine has less, the batched approach is needed.

### 14.5 Timestamps are off by a few hours

Cause: timezone confusion. AIT-ADS timestamps are UTC; the parser explicitly converts to UTC. If downstream tools (Pandas, SQL) interpret naive datetimes as local time, you'll see a shift.

Fix: always treat the parser output as UTC. In Pandas:

```python
df = pd.read_parquet("data/processed/alerts.parquet")
df["timestamp"] = pd.to_datetime(df["timestamp"], utc=True)
```

### 14.6 Suricata severities all look wrong

Cause: forgot the severity inversion. Verify with:

```python
import pandas as pd
df = pd.read_parquet("data/processed/alerts.parquet")
suricata = df[df["source_ids"] == "suricata"]
print(suricata.groupby("severity_raw")["severity_norm"].agg(["min", "max", "mean"]))
```

Expected: severity_raw=1 → severity_norm=5, severity_raw=3 → severity_norm=2. If you see severity_raw=1 → severity_norm=1, the inversion got lost.

### 14.7 Many records show `is_attack=False` and `attack_phase=None`

This is **expected and correct** at this stage. The parser does not apply ground-truth labels. Labelling is a separate step that joins each alert with `labels.csv`. See Section 16.

## 15. Performance Notes

Approximate timings on a mid-range laptop (Intel i5, SSD):

| Step | Time |
|---|---|
| Smoke test (`shaw` only) | ~20 seconds |
| Full ingestion (no parquet) | ~12 minutes |
| Full ingestion (with parquet) | ~15 minutes |
| Reading `alerts.jsonl` line-by-line | ~30 seconds |
| Reading `alerts.parquet` into pandas | ~5 seconds |
| Reading `alerts.csv` into pandas | ~45 seconds |

The bottleneck is JSON parsing, not I/O. If you need it faster, consider replacing `json.loads` with `orjson.loads` — typically a 3–5× speedup for free.

## 16. What Comes After This

The parser is **step 0** of the dissertation pipeline. Here's the planned sequence:

### 16.1 Ground-truth labelling

Walk every alert in `alerts.jsonl`, join against `labels.csv`, and set `is_attack` and `attack_phase` based on whether the alert's timestamp falls within an attack window for that scenario.

`labels.csv` has four columns: `scenario`, `attack`, `start` (Unix epoch seconds), `end` (Unix epoch seconds). Ten attack phases per scenario:

```
network_scans, service_scans, dirb, wpscan, webshell,
cracking, reverse_shell, privilege_escalation, service_stop, dnsteal
```

These map cleanly to MITRE ATT&CK tactics, which is useful for the RAG knowledge base.

### 16.2 Knowledge base construction

Build the RAG retrieval store from:

- CVE descriptions (NIST NVD)
- MITRE ATT&CK technique definitions
- Runbook entries (synthetic, authored from public SRE playbooks)
- Historical incident summaries

Embed with `sentence-transformers/all-MiniLM-L6-v2`, store in ChromaDB.

### 16.3 Train/test splits

Scenario-level splits (train on five scenarios, test on three) to force generalisation across attack patterns rather than memorisation. Avoids the temporal/scenario leakage that random splits would introduce.

### 16.4 Baseline systems

- **Rule-based baseline**: severity thresholds + simple correlation rules
- **LLM-only baseline**: Mistral-7B with no retrieval (isolates the contribution of RAG)

### 16.5 The RAG-enhanced LLM system

Mistral-7B + ChromaDB + the unified knowledge base. The novel contribution.

### 16.6 Experiments

Three-way ablation: rules vs LLM-only vs LLM+RAG. Run on AIT-ADS as primary, HDFS as secondary (for comparability with DeepLog/LogBERT/LogGPT), optionally CAM-LDS for cross-dataset generalisation.

### 16.7 Evaluation

- Quantitative: precision, recall, F1, false positive rate, per attack class
- Qualitative: expert evaluation of explanation quality (Likert scale, Cohen's kappa for inter-rater reliability)
- Operational: latency, throughput, GPU memory footprint

### 16.8 Write-up

Findings, limitations, deployment considerations.
